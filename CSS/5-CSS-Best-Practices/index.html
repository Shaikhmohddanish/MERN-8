Organization and Naming Conventions

BEM Methodology

Standard Card

This is a standard card using BEM naming convention.

Featured Card

This is a featured card with modifier class.


/* Block */
.card {
  background-color: white;
  border-radius: 8px;
  box-shadow: 0 2px 5px rgba(0, 0, 0, 0.1);
  overflow: hidden;
}

/* Elements */
.card__header {
  padding: 1rem;
  border-bottom: 1px solid #eee;
}

.card__title {
  margin: 0;
  font-size: 1.2rem;
}

.card__body {
  padding: 1rem;
}

.card__footer {
  padding: 1rem;
  border-top: 1px solid #eee;
}

.card__button {
  padding: 0.5rem 1rem;
  background-color: #f0f0f0;
  border: none;
  border-radius: 4px;
  cursor: pointer;
}

/* Modifiers */
.card--featured {
  border: 2px solid gold;
}

.card__button--primary {
  background-color: #0066cc;
  color: white;
}

BEM (Block, Element, Modifier) is a naming convention that creates clear, strict relationships between HTML and CSS. The naming follows the pattern block__element--modifier.

CSS Architecture

ITCSS (Inverted Triangle CSS)

Settings
Tools
Generic
Elements
Objects
Components
Utilities

/* 1. Settings - Variables and config */
:root {
  --primary-color: #0066cc;
  --font-size-base: 16px;
}

/* 2. Tools - Mixins and functions */
@mixin center() {
  display: flex;
  align-items: center;
  justify-content: center;
}

/* 3. Generic - Reset and normalize */
* {
  box-sizing: border-box;
  margin: 0;
  padding: 0;
}

/* 4. Elements - Bare HTML elements */
h1 {
  font-size: 2rem;
  margin-bottom: 1rem;
}

/* 5. Objects - Class-based selectors for patterns */
.o-container {
  max-width: 1200px;
  margin: 0 auto;
  padding: 0 15px;
}

/* 6. Components - Specific UI components */
.c-button {
  padding: 0.5rem 1rem;
  background-color: var(--primary-color);
  color: white;
}

/* 7. Utilities - Helpers and overrides */
.u-text-center {
  text-align: center !important;
}

ITCSS organizes CSS by specificity and reach, from broad (settings) to narrow (utilities). This helps manage the cascade and avoid specificity issues.

Atomic CSS uses small, single-purpose classes. Instead of creating semantic classes for each component, you apply multiple utility classes directly to HTML elements.

Performance Optimization

Selector Efficiency

Inefficient Selectors

/* Descendant selectors with high depth */
.header .navigation ul li a {
  color: blue;
}

/* Universal selector with descendant */
.content * {
  line-height: 1.5;
}

/* Inefficient child selector */
div > div > div > p {
  margin: 10px;
}

Efficient Selectors

/* Direct class instead of deep nesting */
.nav-link {
  color: blue;
}

/* Targeted element selection */
.content-text {
  line-height: 1.5;
}

/* Simplified selector */
.deep-paragraph {
  margin: 10px;
}

CSS selectors are read from right to left by browsers. Simpler, more specific selectors are more efficient than complex, deeply nested ones.

Reflows and Repaints

Causes Reflow (Expensive)

Reflow Animation
.reflow-element {
  position: relative;
  left: 0;
  top: 0;
  transition: left 1s, top 1s;
}

.reflow-element.animate {
  left: 100px;
  top: 50px;
}

Uses Transform (Efficient)

Transform Animation
.transform-element {
  transition: transform 1s;
}

.transform-element.animate {
  transform: translate(100px, 50px);
}

Changes to layout properties (like width, height, position) cause reflows, which are computationally expensive. Using transform and opacity for animations is more efficient.

CSS Containment

Without Containment

Changes here can cause full document reflow

With Containment

Changes here are isolated


/* Containing element updates to improve performance */
.contain-strict {
  contain: content;
  /* or more specific: */
  contain: layout paint;
}

/* This tells the browser that the internal layout of this 
   element is independent of the rest of the page, allowing 
   the browser to optimize rendering. */

The contain property informs the browser that an element's content is independent of the rest of the page, enabling performance optimizations.

Vendor prefixes ensure compatibility with older browsers. Tools like Autoprefixer can automatically add these prefixes based on browser support data.

Feature queries (@supports) let you apply styles conditionally based on browser support for specific CSS features.

Responsive Variables

Resize the browser to see changes

This element uses responsive variables

:root {
  /* Base variables */
  --spacing-unit: 8px;
  --font-size-base: 16px;
  --container-width: 100%;
}

/* Tablet and up */
@media (min-width: 768px) {
  :root {
    --spacing-unit: 12px;
    --font-size-base: 18px;
    --container-width: 750px;
  }
}

/* Desktop and up */
@media (min-width: 992px) {
  :root {
    --spacing-unit: 16px;
    --font-size-base: 20px;
    --container-width: 970px;
  }
}

/* Using the variables */
.variable-box {
  max-width: var(--container-width);
  margin: 0 auto;
  padding: calc(var(--spacing-unit) * 2);
}

.variable-item {
  font-size: var(--font-size-base);
  padding: var(--spacing-unit);
  background-color: #d1ecf1;
  border-radius: calc(var(--spacing-unit) / 2);
}

You can change CSS variable values within media queries to create responsive designs that adapt to different screen sizes.

CSS Preprocessors

Sass Features

Variables

$primary-color: #0066cc;
$spacing: 16px;

.button {
  background: $primary-color;
  padding: $spacing;
}

Nesting

.navbar {
  background: #333;
  
  ul {
    margin: 0;
    
    li {
      display: inline-block;
      
      a {
        color: white;
        
        &:hover {
          text-decoration: underline;
        }
      }
    }
  }
}

Mixins

@mixin flex-center {
  display: flex;
  justify-content: center;
  align-items: center;
}

.centered-box {
  @include flex-center;
  height: 200px;
}

Functions

@function calculateRem($size) {
  $remSize: $size / 16px;
  @return $remSize * 1rem;
}

h1 {
  font-size: calculateRem(32px);
}

Sass is a CSS preprocessor that adds features like variables, nesting, mixins, and functions to make CSS more maintainable and powerful.

Output Comparison

Sass/SCSS Input

$primary: #0066cc;
$secondary: #ff6600;
$padding-base: 15px;

@mixin box-shadow($opacity) {
  box-shadow: 0 2px 5px rgba(0, 0, 0, $opacity);
}

.button {
  padding: $padding-base;
  
  &.primary {
    background-color: $primary;
    color: white;
    @include box-shadow(0.3);
  }
  
  &.secondary {
    background-color: $secondary;
    color: white;
    @include box-shadow(0.2);
  }
}

Compiled CSS Output

.button {
  padding: 15px;
}

.button.primary {
  background-color: #0066cc;
  color: white;
  box-shadow: 0 2px 5px rgba(0, 0, 0, 0.3);
}

.button.secondary {
  background-color: #ff6600;
  color: white;
  box-shadow: 0 2px 5px rgba(0, 0, 0, 0.2);
}

CSS preprocessors compile to standard CSS that browsers can understand. The compiled output is what gets served to the browser.

CSS Frameworks

Framework Comparison

Framework Approach File Size Learning Curve Customization

CSS frameworks provide pre-built components and styles. Each has its own approach, advantages, and trade-offs.

Component-based frameworks like Bootstrap use semantic class names, while utility-first frameworks like Tailwind CSS use atomic classes that describe styles directly.

When to Use a Framework

Pros of Using a Framework

  • Faster development time
  • Consistent UI components
  • Built-in responsive design
  • Cross-browser compatibility
  • Community support and documentation

Cons of Using a Framework

  • Increased file size
  • Generic design (unless heavily customized)
  • Learning curve for framework-specific classes
  • Potential for unused CSS
  • Less control over implementation details

Consider using a framework for rapid prototyping, consistent UI, or when working with a team. For unique designs or performance-critical sites, a custom approach might be better.
